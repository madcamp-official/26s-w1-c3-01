# 먹픽 기능명세서

## 1. 프로젝트 개요

### 1.1 프로젝트명

먹픽 (MUK PICK)

### 1.2 프로젝트 주제

개인 및 집단의 음식 선호도, 제한 조건, 최근 식사 기록, 예산 조건을 반영하여 메뉴 결정 갈등을 줄여주는 메뉴 추천 앱

### 1.3 프로젝트 목적

먹픽은 개인과 단체가 식사 메뉴를 더 빠르고 만족스럽게 결정할 수 있도록 돕는 것을 목적으로 한다. 개인 사용자는 일상적인 식사 상황에서 자신의 음식 취향, 알러지 및 제한 조건, 최근 식사 기록, 예산 조건을 기반으로 메뉴를 추천받을 수 있다.

단체 사용자는 약속이나 모임에서 서로 다른 취향으로 인해 발생하는 메뉴 선택 갈등을 줄일 수 있다. 먹픽은 참여자의 선호도와 제한 조건을 종합하여 특정 사람에게 지나치게 불리하지 않고 모임 전체가 수용하기 쉬운 메뉴 후보를 추천한다.

먹픽은 단순한 랜덤 메뉴 추천 방식이 아니라, 정형화된 사용자 선호도와 메뉴 feature를 활용한 점수 기반 추천 방식을 사용한다. 개인 추천에서는 카테고리 선호도, 태그 선호도, 메뉴 선호도, 예산 조건, 최근 식사 기록을 반영하고, 모임 추천에서는 참여자별 선호도와 모임 목적 적합도를 함께 반영한다.

### 1.4 기대 효과

* 메뉴 선택 과정에서 불필요한 고민과 대기 시간을 줄일 수 있다.
* 개인의 음식 취향과 최근 식사 이력이 추천 결과에 반영되어 사용자 만족도를 높일 수 있다.
* 예산 조건을 반영해 현실적인 메뉴 후보를 추천할 수 있다.
* 단체 모임에서 특정 사람의 의견에 치우치지 않는 균형 잡힌 메뉴 선택이 가능하다.
* 구성원 간 선호도 차이로 인한 갈등을 줄이고 의사결정 과정을 원활하게 만들 수 있다.
* 익숙한 메뉴뿐만 아니라 새로운 메뉴 후보를 제안하여 식사 선택의 다양성을 높일 수 있다.
* 좋아요, 싫어요, 북마크와 같은 메뉴 상호작용을 통해 사용자 경험을 개선할 수 있다.

## 2. 예상 사용자

### 2.1 개인 사용자

혼자 식사할 때 어떤 메뉴를 먹을지 결정하기 어려운 사용자이다. 자신의 선호도, 제한 조건, 예산, 최근 식사 기록을 기반으로 메뉴를 추천받을 수 있다.

### 2.2 회원 사용자

이메일/비밀번호 또는 Google OAuth를 통해 로그인하는 사용자이다. 자신의 선호도, 식사 기록, 메뉴 상호작용, 모임 참여 정보를 지속적으로 관리할 수 있다.

### 2.3 모임 생성자

친구, 팀원, 동료 등과의 식사 약속을 만들고 참여자를 초대하는 사용자이다. 모임의 제목, 날짜, 시간, 식사 목적 등을 설정하고, 모임 메뉴 추천을 요청할 수 있다. 모임 생성자는 추천 계산에 포함할 참여자를 선택할 수 있으며, 추천 결과 중 하나를 최종 메뉴로 확정할 수 있다.

### 2.4 모임 참여자

생성된 모임에 참여하여 자신의 음식 선호도, 제한 조건, 알러지, 예산 조건 등을 추천에 반영하는 사용자이다. 추천된 메뉴 후보와 추천 이유를 확인할 수 있다.

### 2.5 게스트 참여자

회원가입 없이 임시 계정으로 모임에 참여하는 사용자이다. 게스트는 선호도와 제한 조건을 입력한 뒤 모임 ID로 참여한다. 게스트의 내부 닉네임은 서버가 랜덤 생성하고, 모임 안에서 보이는 이름은 모임 참여 시 입력하는 표시 이름을 사용한다. 게스트는 모임 추천 결과를 확인할 수 있지만, 최종 메뉴 확정은 할 수 없다.

## 3. 전체 서비스 흐름

### 3.1 이메일 회원가입 및 로그인 흐름

1. 사용자가 이메일, 비밀번호, 닉네임을 입력한다.
2. 시스템은 이메일 형식, 비밀번호 조건, 닉네임 중복 여부를 검사한다.
3. 사용자는 이메일 회원가입을 완료한다.
4. 필요한 경우 인증 이메일 재전송을 요청할 수 있다.
5. 사용자는 이메일과 비밀번호로 로그인한다.
6. 로그인 성공 시 시스템은 access token, refresh token, 만료 시각을 저장한다.
7. 로그인 후 사용자는 앱 메인 화면으로 진입한다.

### 3.2 Google OAuth 로그인 흐름

1. 사용자가 Google 로그인 버튼을 누른다.
2. 시스템은 Supabase OAuth를 통해 Google 인증을 진행한다.
3. OAuth callback 이후 access token과 refresh token을 저장한다.
4. 시스템은 백엔드에 프로필 동기화를 요청한다.
5. 신규 사용자이거나 닉네임 입력이 필요한 경우 닉네임 온보딩 화면으로 이동한다.
6. 사용자가 닉네임을 입력하면 닉네임 중복 검사를 수행한다.
7. 닉네임 설정이 완료되면 사용자 프로필을 동기화하고 앱 메인 화면으로 이동한다.

### 3.3 게스트 모임 참여 흐름

1. 사용자가 게스트 시작을 선택한다.
2. 시스템은 임시 게스트 계정을 생성한다.
3. 게스트는 음식 카테고리, 태그, 알러지 및 제한 조건 등 기본 선호도를 입력한다.
4. 게스트는 모임 ID를 입력한다.
5. 시스템은 모임 preview 정보를 조회하여 모임 제목과 기존 구성원을 보여준다.
6. 게스트는 모임 안에서 사용할 표시 이름을 입력한다.
7. 시스템은 표시 이름이 같은 모임 안에서 중복되지 않는지 확인한다.
8. 게스트는 모임에 참여한다.
9. 게스트는 모임 상세와 추천 결과를 확인할 수 있다.
10. 메뉴가 확정되면 backend는 게스트 임시 계정을 정리할 수 있고, 참여 기록에는 표시 이름만 남는다.

### 3.4 개인 메뉴 추천 흐름

1. 사용자가 앱에 로그인한다.
2. 사용자가 음식 선호도, 알러지 및 제한 조건, 예산 조건, 최근 식사 중복 패널티 조건을 설정한다.
3. 사용자가 개인 메뉴 추천 기능을 선택한다.
4. 시스템은 사용자의 카테고리 선호도, 태그 선호도, 메뉴 선호도, 알러지, 예산, 식사 기록을 확인한다.
5. 시스템은 알러지와 충돌하는 메뉴를 제외한다.
6. 시스템은 최근에 먹은 메뉴에 대해 중복 패널티를 적용한다.
7. 시스템은 적합한 메뉴 후보를 점수순으로 추천한다.
8. 사용자는 추천 메뉴 중 하나를 선택한다.
9. 사용자가 추천 메뉴를 확정하면 해당 메뉴가 식사 기록에 저장된다.

### 3.5 모임 메뉴 추천 흐름

1. 모임 생성자가 식사 약속을 생성한다.
2. 모임 생성자는 모임 ID를 공유한다.
3. 참여자는 모임 ID를 입력해 모임 정보와 기존 구성원을 확인한다.
4. 참여자는 모임 안에서 사용할 표시 이름을 입력한다.
5. 참여자는 자신의 선호도와 제한 조건을 선택한다.
6. 시스템은 참여자들의 선호도를 종합하여 모임 선호도를 분석한다.
7. 모임 생성자는 구성원 칩을 켜고 끄며 추천 계산에 포함할 사람을 선택할 수 있다.
8. 시스템은 선택된 참여자들의 선호도와 모임 목적 적합도를 기준으로 메뉴 후보를 랭킹으로 추천한다.
9. 참여자는 추천 결과와 추천 이유를 확인한다.
10. 모임 생성자가 추천 메뉴 중 하나를 최종 확정한다.
11. 메뉴가 확정되면 모임 상태는 확정 상태로 바뀌고, 게스트 임시 계정은 정리될 수 있다.

## 4. 핵심 기능 목록

먹픽의 핵심 기능은 사용자 관리, 개인 선호도 관리, 개인 메뉴 추천, 모임 관리, 모임 메뉴 추천, 식사 기록 관리, 메뉴 상호작용, 메뉴 데이터 관리 기능으로 구분한다.

### 4.1 사용자 관리 기능

| 기능 ID | 기능명              | 설명                                                                  |
| ----- | ---------------- | ------------------------------------------------------------------- |
| U-01  | 이메일 회원가입         | 사용자가 이메일, 비밀번호, 닉네임을 입력하여 계정을 생성한다.                                 |
| U-02  | 로그인 및 로그아웃       | 사용자가 계정 정보를 입력하여 서비스에 접속하고, 사용을 마친 뒤 계정에서 로그아웃한다.                   |
| U-03  | Google OAuth 로그인 | 사용자가 Google 계정을 이용해 로그인하고, 신규 사용자는 닉네임 온보딩을 거친다.                    |
| U-04  | 닉네임 중복 확인        | 회원가입 또는 OAuth 온보딩 중 닉네임 중복 여부를 확인한다.                                |
| U-05  | 사용자 정보 관리        | 사용자가 닉네임, 기본 정보, 계정 정보를 확인하거나 수정한다.                                 |
| U-06  | 게스트 시작           | 사용자가 회원가입 없이 임시 계정을 만들고 선호도 입력 후 모임에 참여한다.                          |
| U-07  | 세션 유지 및 만료 처리    | access token과 refresh token을 사용해 로그인 상태를 유지하고, 만료 시 다시 로그인하도록 안내한다. |

| 기능 ID | 기능명             | 설명                                             |
| ----- | --------------- | ---------------------------------------------- |
| P-01  | 카테고리 선호도 선택     | 사용자가 한식, 중식, 양식, 일식 등 음식 카테고리 선호도를 선택한다.       |
| P-02  | 태그 선호도 선택       | 사용자가 구이, 국물, 볶음, 튀김, 매운맛 등 음식 특성 태그 선호도를 선택한다. |

| P-04  | 선호 점수 관리        | 사용자가 카테고리와 태그에 대해 0~5 범위의 선호 점수를 설정한다.         |
| P-05  | 예산 조건 설정        | 사용자가 선호 예산 범위를 설정한다.                           |
| P-06  | 최근 식사 중복 패널티 설정 | 사용자가 최근 며칠 이내에 먹은 메뉴를 감점할지 설정한다.               |

| R-03  | 새로운 메뉴 포함 여부 선택 | 사용자가 기존 식사 기록에 없던 새로운 메뉴를 추천 후보에 포함할지 선택한다.                          |

| R-06  | 개인 추천 재조회       | 사용자가 조건을 바꾸거나 다시 추천받고 싶을 때 개인 추천 결과를 재조회한다.                          |

## 5. 기능별 상세 명세

### 5.1 사용자 관리 기능

* **입력**

  * 이메일
  * 비밀번호
  * 닉네임
  * Google OAuth 인증 정보
  * 게스트 표시 이름
  * 수정할 사용자 기본 정보

* **처리**

  * 사용자는 이메일 회원가입을 통해 계정을 생성한다.
  * 사용자는 이메일과 비밀번호를 사용해 로그인하거나 로그아웃할 수 있다.
  * 사용자는 Google OAuth를 통해 로그인할 수 있다.
  * OAuth 신규 사용자는 닉네임 입력 온보딩을 진행한다.
  * 시스템은 회원가입 및 OAuth 온보딩 시 닉네임 중복 여부를 확인한다.
  * 시스템은 로그인 성공 시 access token, refresh token, 만료 시각을 저장한다.
  * access token이 만료되기 전에 refresh token으로 세션을 갱신한다.
  * refresh 실패 또는 인증 오류가 발생하면 저장된 세션을 삭제하고 다시 로그인하도록 안내한다.
  * 게스트 사용자는 임시 계정을 생성하고, 모임 참여 시 표시 이름을 입력한다.
  * 사용자가 닉네임 등 기본 정보를 수정할 경우 변경된 정보의 유효성을 확인한 뒤 저장한다.

* **출력**

  * 회원가입, 로그인, OAuth 로그인, 로그아웃, 사용자 정보 수정 결과를 메시지로 제공한다.
  * 처리 결과에 따라 시작 화면, 로그인 화면, 닉네임 온보딩 화면, 메인 화면, 사용자 정보 화면 등으로 이동한다.
  * 게스트 참여자는 모임 화면으로 이동한다.

* **예외 상황**

  * 이메일 형식이 올바르지 않은 경우 오류 메시지를 표시한다.
  * 이미 가입된 이메일인 경우 오류 메시지를 표시한다.
  * 비밀번호가 조건에 맞지 않는 경우 다시 입력하도록 안내한다.
  * 닉네임이 중복된 경우 다른 닉네임을 입력하도록 안내한다.
  * 로그인 정보가 일치하지 않는 경우 다시 입력하도록 안내한다.
  * OAuth 프로필 동기화에 실패하면 다시 시도하도록 안내한다.
  * refresh token이 만료되었거나 유효하지 않으면 세션을 정리하고 다시 로그인하도록 안내한다.
  * 사용자 정보 수정에 실패한 경우 저장되지 않았음을 안내한다.

### 5.2 개인 선호도 관리 기능

* **입력**

  * 음식 카테고리
  * 음식 태그
  * 카테고리별 선호 점수
  * 태그별 선호 점수
  * 알러지 재료
  * 제한 조건
  * 예산 조건
  * 최근 식사 중복 패널티 기간
  * 새로운 메뉴 포함 여부

* **처리**

  * 사용자는 앱에서 제공하는 항목 중 자신의 음식 취향에 맞는 조건을 선택한다.
  * 사용자는 카테고리와 태그에 대해 0~5 범위의 선호 점수를 설정한다.
  * 사용자는 알러지 재료, 제한 조건 등 추천에서 제외해야 할 조건을 선택한다.
  * 사용자는 선호 예산 범위를 설정할 수 있다.
  * 사용자는 최근 며칠 이내에 먹은 메뉴를 추천에서 감점할지 설정할 수 있다.
  * 시스템은 선택된 선호도와 제한 조건을 사용자 프로필에 저장한다.
  * 저장된 선호도 정보는 개인 메뉴 추천과 모임 메뉴 추천에 활용된다.
  * 사용자는 저장된 선호도 정보를 조회하거나 수정할 수 있다.

* **출력**

  * 선택된 음식 선호도 정보를 표시한다.
  * 선택된 알러지 정보와 제한 조건 목록을 표시한다.
  * 예산 조건과 최근 식사 중복 패널티 설정값을 표시한다.
  * 선호도 설정 또는 수정이 완료되면 저장 완료 메시지를 제공한다.

* **예외 상황**

  * 사용자가 선호도 정보를 선택하지 않은 경우 기본 추천 조건을 사용하거나 선호도 설정을 안내한다.
  * 알러지 등 필수 제한 조건이 있는 경우 해당 조건과 충돌하는 메뉴는 추천에서 제외한다.
  * 예산 하한이 상한보다 큰 경우 저장할 수 없도록 안내한다.
  * 선호도 저장 과정에서 오류가 발생하면 다시 시도하도록 안내한다.

### 5.3 개인 메뉴 추천 기능

* **입력**

  * 사용자의 음식 카테고리 선호도
  * 사용자의 태그 선호도
  * 사용자의 메뉴 선호도
  * 알러지 및 제한 조건
  * 예산 조건
  * 최근 식사 기록
  * 최근 식사 중복 패널티 기간
  * 새로운 메뉴 포함 여부
  * 추천 결과 개수

* **처리**

  * 사용자가 개인 메뉴 추천을 요청하면 시스템은 사용자의 선호도와 제한 조건을 확인한다.
  * 시스템은 추천 가능한 메뉴 후보를 탐색한다.
  * 알러지 조건과 충돌하는 메뉴는 후보에서 제외한다.
  * 카테고리 선호도, 태그 선호도, 메뉴 선호도, 예산 적합도를 점수화한다.
  * 최근에 먹은 메뉴는 설정된 기간에 따라 감점한다.
  * 새로운 메뉴 포함 여부가 꺼져 있으면 식사 기록에 없는 메뉴를 후보에서 제외한다.
  * 시스템은 점수가 높은 메뉴를 우선 추천한다.
  * 추천 실행 결과와 세부 점수를 저장한다.

* **출력**

  * 개인에게 적합한 추천 메뉴 후보를 제공한다.
  * 각 메뉴의 순위, 점수, 추천 이유를 제공한다.
  * 새로운 메뉴 여부를 표시할 수 있다.
  * 사용자가 선택한 추천 메뉴를 확정할 수 있도록 한다.

* **예외 상황**

  * 추천 가능한 메뉴가 부족한 경우 일부 조건을 완화하거나 다른 카테고리 선택을 안내한다.
  * 사용자의 선호도 정보가 부족한 경우 기본 메뉴 후보를 제공하거나 선호도 설정을 먼저 진행하도록 안내한다.
  * 알러지 조건과 충돌하는 메뉴는 추천 결과에 표시하지 않는다.
  * 추천 생성 중 오류가 발생하면 다시 시도하도록 안내한다.

### 5.4 모임 관리 기능

* **입력**

  * 모임 이름
  * 날짜
  * 시간
  * 모임 목적
  * 참여자 정보
  * 모임에서 사용할 표시 이름
  * 수정할 모임 정보

* **처리**

  * 모임 생성자는 식사 약속을 생성하고 모임의 기본 정보를 설정한다.
  * 시스템은 생성된 모임 정보를 저장한다.
  * 시스템은 모임 ID를 통해 참여자를 초대할 수 있게 한다.
  * 참여자는 모임 ID로 모임 preview 정보를 확인할 수 있다.
  * 참여자는 모임 안에서 사용할 표시 이름을 입력하고 모임에 참여한다.
  * 표시 이름은 같은 모임 안에서 중복될 수 없다.
  * 사용자는 모임 정보, 참여자 목록, 추천 진행 상태를 확인할 수 있다.
  * 모임 생성자는 모임 정보를 수정할 수 있다.
  * 게스트 사용자는 모임 참여 후 모임 화면을 중심으로 서비스를 이용한다.

* **출력**

  * 모임 생성 완료 메시지를 표시한다.
  * 모임 ID와 초대 정보를 제공한다.
  * 모임 상세 정보, 참여자 목록, 추천 진행 상태를 표시한다.
  * 게스트에게는 모임 preview와 참여 결과를 제공한다.

* **예외 상황**

  * 모임 이름, 날짜, 시간, 목적 등 필수 정보가 누락된 경우 모임 생성을 제한한다.
  * 존재하지 않는 모임 ID를 입력한 경우 오류 메시지를 표시한다.
  * 같은 모임 안에서 표시 이름이 중복된 경우 다른 이름을 입력하도록 안내한다.
  * 초대받은 사용자가 참여하지 않은 경우 해당 사용자를 제외하고 추천을 진행하거나 대기 상태로 표시한다.
  * 모임 정보 수정 권한이 없는 사용자가 수정을 시도할 경우 수정할 수 없도록 안내한다.

### 5.5 모임 메뉴 추천 기능

* **입력**

  * 모임 참여자들의 음식 선호도
  * 모임 참여자들의 알러지 및 제한 조건
  * 모임 참여자들의 메뉴 선호도
  * 모임 참여자들의 예산 조건
  * 모임 목적
  * 추천 계산에 포함할 참여자 목록
  * 추천 결과 개수

* **처리**

  * 시스템은 모임 참여자들이 설정한 선호도와 제한 조건을 종합한다.
  * 모임 생성자가 제외한 구성원은 추천 계산에서 제외한다.
  * 시스템은 해당 모임의 목적에 적합한 메뉴를 확인한다.
  * 참여자의 알러지와 충돌하는 메뉴는 후보에서 제외한다.
  * 참여자 중 한 명이라도 특정 메뉴 또는 카테고리를 명시적으로 0점으로 설정한 경우 해당 메뉴는 후보에서 제외한다.
  * 참여자별 카테고리, 태그, 메뉴, 예산 점수를 계산한다.
  * 참여자별 점수 평균과 모임 목적 적합도 점수를 합산한다.
  * 특정 참여자에게 지나치게 불리한 메뉴가 상위에 오르지 않도록 최소 참여자 점수를 정렬 기준에 반영한다.
  * 추천 실행 결과와 세부 점수를 저장한다.
  * 추천 결과가 생성되면 모임 상태를 추천 완료 상태로 변경한다.

* **출력**

  * 모임에 적합한 메뉴 후보 목록을 제공한다.
  * 각 메뉴의 순위, 점수, 추천 이유를 함께 제공한다.
  * 추천 결과는 모임 참여자들이 확인할 수 있도록 모임 화면에 표시한다.
  * 각 메뉴에는 랭킹, 점수, 추천 이유, 선택 상태, 확정 버튼을 표시한다.
  * 모임 생성자는 추천 후보 중 하나를 최종 메뉴로 확정할 수 있다.

* **예외 상황**

  * 참여자들의 선호 조건이 크게 충돌하는 경우 공통적으로 제외해야 하는 조건을 우선 반영한다.
  * 선호도 충돌이 큰 경우 비교적 중립적인 메뉴 후보를 제안한다.
  * 참여자 정보가 부족한 경우 현재까지 입력된 정보를 기준으로 추천을 제공한다.
  * 모든 조건을 만족하는 메뉴가 없는 경우 조건 완화가 필요함을 안내한다.
  * 게스트가 메뉴 확정을 시도할 경우 확정할 수 없도록 제한한다.
  * 최신 추천 결과에 없는 메뉴를 확정하려는 경우 확정을 제한한다.

### 5.6 식사 기록 관리 기능

* **입력**

  * 개인 추천 또는 모임 추천을 통해 선택된 메뉴
  * 식사 날짜
  * 만족도
  * 식사 메모
  * 수정할 식사 기록 정보

* **처리**

  * 시스템은 사용자가 확정한 메뉴를 식사 기록으로 저장한다.
  * 개인 추천에서 선택된 메뉴는 사용자가 확정한 뒤에만 식사 기록으로 저장한다.
  * 사용자는 이전 식사 기록을 조회할 수 있다.
  * 사용자는 식사 기록의 날짜, 만족도, 메모를 수정할 수 있다.
  * 사용자는 불필요한 식사 기록을 삭제할 수 있다.
  * 식사 기록의 평점은 이후 메뉴 선호도 보정에 활용할 수 있다.
  * 최근 식사 기록은 개인 추천의 중복 패널티 계산에 활용된다.

* **출력**

  * 사용자의 이전 선택 메뉴를 제공한다.
  * 식사 날짜, 메뉴명, 만족도, 메모를 표시한다.
  * 메뉴 선택이나 만족도 저장이 완료되면 저장 완료 메시지를 표시한다.
  * 수정 또는 삭제가 완료되면 화면 상태를 갱신한다.

* **예외 상황**

  * 기록 저장에 실패한 경우 저장되지 않았음을 안내하고 다시 시도할 수 있도록 한다.
  * 기록이 없는 신규 사용자의 경우 조회할 기록이 없음을 표시한다.
  * 만족도를 입력하지 않은 경우 선택 메뉴와 날짜 중심으로 저장한다.
  * 수정 또는 삭제 권한이 없는 기록은 변경할 수 없도록 한다.

### 5.7 메뉴 상호작용 기능

* **입력**

  * 메뉴 ID
  * 상호작용 유형
  * 선택 여부

* **처리**

  * 사용자는 메뉴에 좋아요, 싫어요, 북마크를 표시할 수 있다.
  * 시스템은 사용자의 메뉴별 상호작용 상태를 저장한다.
  * 좋아요와 싫어요는 동시에 활성화되지 않도록 처리한다.
  * 북마크는 좋아요/싫어요와 별도로 유지할 수 있다.
  * 사용자는 기존 상호작용 상태를 다시 눌러 해제할 수 있다.
  * 메뉴 조회, 선택 등 행동 로그를 저장할 수 있다.

* **출력**

  * 메뉴별 좋아요, 싫어요, 북마크 상태를 표시한다.
  * 상호작용 변경이 완료되면 화면 상태를 즉시 갱신한다.

* **예외 상황**

  * 존재하지 않는 메뉴에 상호작용을 시도할 경우 오류 메시지를 표시한다.
  * 로그인하지 않은 사용자가 상호작용을 시도할 경우 로그인 또는 게스트 시작을 안내한다.
  * 상호작용 저장에 실패하면 이전 상태로 되돌리거나 다시 시도하도록 안내한다.

### 5.8 메뉴 데이터 관리 기능

* **입력**

  * 메뉴명
  * 음식 카테고리
  * 메뉴 설명
  * 매운 정도
  * 가격대
  * 칼로리
  * 메뉴 태그
  * 알러지 관련 정보
  * 모임 목적 적합도
  * 메뉴 이미지 또는 UI 자산

* **처리**

  * 시스템은 추천에 사용할 메뉴 데이터를 등록하고 관리한다.
  * 각 메뉴를 한식, 중식, 일식, 양식, 아시안, 고기, 패스트푸드 등 음식 카테고리별로 분류한다.
  * 메뉴별 매운 정도, 가격대, 칼로리, 태그 등의 속성 정보를 관리한다.
  * 알러지나 제한 조건과 관련된 정보를 함께 관리한다.
  * 모임 목적별 메뉴 적합도 점수를 관리한다.
  * 메뉴 이미지나 배경 이미지 등 UI 자산은 Supabase Storage 또는 프론트 자산으로 관리한다.
  * 부적절한 메뉴가 추천되지 않도록 추천 제외 조건을 반영한다.

* **출력**

  * 추천에 활용 가능한 메뉴 목록을 제공한다.
  * 카테고리 정보를 제공한다.
  * 태그 정보를 제공한다.
  * 알러지 및 제한 조건 정보를 제공한다.
  * 모임 목적 정보를 제공한다.
  * 메뉴 속성 정보를 제공한다.

* **예외 상황**

  * 메뉴 데이터가 부족한 경우 추천 결과가 제한될 수 있음을 안내한다.
  * 메뉴 속성이나 알러지 정보가 누락된 경우 추천 정확도가 낮아질 수 있음을 표시한다.
  * 중복 메뉴가 등록될 경우 기존 데이터와 비교하여 중복 여부를 확인한다.
  * 이미지 또는 UI 자산 로딩에 실패한 경우 fallback 이미지를 사용한다.

## 6. 추천 로직 기준

### 6.1 개인 추천 기준

개인 메뉴 추천은 다음 요소를 반영한다.

* 사용자 카테고리 선호도
* 사용자 태그 선호도
* 사용자 메뉴 선호도
* 식사 기록 평점
* 사용자 알러지 및 제한 조건
* 메뉴 가격대와 사용자 예산 조건
* 최근 식사 기록
* 새로운 메뉴 포함 여부

개인 추천에서 알러지와 충돌하는 메뉴는 점수 감점 방식이 아니라 후보에서 제외한다.

### 6.2 개인 추천 점수식

개인 추천 점수는 최대 100점이다.

```text
totalScore = categoryScore + tagScore + menuPreferenceScore + budgetScore - historyPenalty
```

세부 점수:

| 항목        |    최대 점수 | 설명                                  |
| --------- | -------: | ----------------------------------- |
| 카테고리 선호도  |       35 | 사용자의 카테고리 선호도 0~5를 35점으로 환산         |
| 태그 선호도    |       25 | 메뉴 태그에 대한 사용자 선호도 평균을 25점으로 환산      |
| 메뉴 선호도    |       25 | 명시적 메뉴 선호도 또는 식사 기록 평점 평균을 25점으로 환산 |
| 예산 적합도    |       15 | 메뉴 가격대가 예산 범위에 맞는지 반영               |
| 최근 식사 패널티 | 최대 35 차감 | 최근에 먹은 메뉴일수록 강하게 감점                 |

예산 점수 기준:

| 조건                       | 점수 |
| ------------------------ | -: |
| 메뉴 가격대가 없거나 예산 상한이 없는 경우 | 15 |
| 메뉴 가격대가 예산 상한보다 높은 경우    |  8 |
| 메뉴 가격대가 예산 하한보다 낮은 경우    | 12 |
| 메뉴 가격대가 예산 범위 안에 있는 경우   | 15 |

최근 식사 패널티 기준:

```text
historyPenalty = 35 * (1 - daysSinceLastEaten / recentDuplicateDays)
```

최근 식사 기록이 없거나 `recentDuplicateDays <= 0`이면 패널티는 적용하지 않는다.

### 6.3 모임 추천 기준

모임 메뉴 추천은 다음 요소를 반영한다.

* 참여자 전체의 공통 선호도
* 참여자별 카테고리 선호도
* 참여자별 태그 선호도
* 참여자별 메뉴 선호도
* 참여자별 예산 조건
* 참여자별 알러지 및 제한 조건
* 모임 목적 적합도
* 특정 참여자에게 지나치게 불리하지 않은 메뉴 여부

모임 추천에서 알러지 충돌, 명시적 메뉴 0점, 명시적 카테고리 0점, 목적 적합도 0점 메뉴는 후보에서 제외한다.

### 6.4 모임 추천 점수식

모임 추천 점수는 최대 100점이다.

```text
participantRawScore = categoryScore + tagScore + menuPreferenceScore + budgetScore
participantPreferenceScore = participantRawScore / 85 * 80
meetingScore = average(participantPreferenceScore) + purposeScore
```

참여자별 점수 구성:

| 항목       | 최대 점수 | 설명                        |
| -------- | ----: | ------------------------- |
| 카테고리 선호도 |    30 | 참여자 카테고리 선호도 반영           |
| 태그 선호도   |    20 | 참여자 태그 선호도 반영             |
| 메뉴 선호도   |    25 | 명시적 메뉴 선호도 또는 식사 기록 평점 반영 |
| 예산 적합도   |    10 | 메뉴 가격대와 참여자 예산 조건 반영      |

모임 목적 점수:

| 항목     | 최대 점수 | 설명                    |
| ------ | ----: | --------------------- |
| 목적 적합도 |    20 | 모임 목적에 맞는 메뉴일수록 높은 점수 |

정렬 기준:

1. 총점이 높은 메뉴
2. 최소 참여자 점수가 높은 메뉴
3. 모임 목적 점수가 높은 메뉴
4. 그룹 평균 선호 점수가 높은 메뉴
5. 메뉴명 가나다순
6. 메뉴 ID 오름차순

## 9. 예외 상황 처리

| 예외 상황                          | 처리 방식                                      |
| ------------------------------ | ------------------------------------------ |
| 사용자가 선호도를 설정하지 않은 경우           | 기본 추천 조건을 사용하거나 선호도 설정을 안내한다.              |
| 알러지 또는 제한 재료가 포함된 메뉴가 있는 경우    | 해당 메뉴는 추천 후보에서 제외한다.                       |
| 예산 조건이 메뉴 가격대와 맞지 않는 경우        | 후보 제외가 아니라 점수 감점 방식으로 처리한다.                |
| 최근에 먹은 메뉴가 추천 후보에 포함되는 경우      | 설정된 기간에 따라 감점한다.                           |
| 모임 참여자 중 일부가 선호 조건을 선택하지 않은 경우 | 입력된 참여자 정보를 기준으로 추천을 제공한다.                 |
| 참여자들의 선호도가 크게 충돌하는 경우          | 공통 제한 조건을 우선 반영하고, 비교적 중립적인 메뉴 후보를 제안한다.   |
| 추천 가능한 메뉴가 부족한 경우              | 조건 완화 또는 다른 카테고리 선택을 안내한다.                 |
| 메뉴 데이터가 부족한 경우                 | 추천 결과가 제한될 수 있음을 안내한다.                     |
| 기록 저장에 실패한 경우                  | 저장 실패 메시지를 표시하고 다시 시도할 수 있도록 안내한다.         |
| token이 만료된 경우                  | refresh token으로 갱신하고, 실패하면 다시 로그인하도록 안내한다. |
| API 서버가 JSON이 아닌 응답을 반환한 경우    | API base URL 또는 백엔드 서버 상태를 확인하도록 안내한다.     |
| 게스트가 메뉴 확정을 시도한 경우             | 확정 권한이 없음을 안내한다.                           |
| 표시 이름이 같은 모임 안에서 중복된 경우        | 다른 표시 이름을 입력하도록 안내한다.                      |
| 메뉴 상호작용 저장에 실패한 경우             | 변경 실패를 안내하고 이전 상태를 유지하거나 재시도하도록 한다.        |

## 10. 비기능 요구사항

### 10.1 사용성

사용자는 복잡한 입력 없이 빠르게 메뉴를 추천받을 수 있어야 한다. 개인 추천은 가능한 한 짧은 단계로 진행되어야 하며, 모임 추천도 초대와 선호도 선택 과정이 간단해야 한다.

선호도 입력 과정에서는 카테고리, 태그, 제한 조건, 예산, 최근 식사 중복 패널티를 단계적으로 입력할 수 있어야 한다. 모바일 화면에서도 버튼 위치, 입력 영역, 추천 결과 카드가 잘리지 않아야 한다.

### 10.2 성능

추천 결과는 사용자가 오래 기다리지 않도록 빠르게 제공되어야 한다. 초기 버전에서는 정형 데이터 기반 점수 계산 방식을 사용하여 안정적인 응답 속도를 확보한다.

프론트엔드는 추천 생성, 모임 확정, 식사 기록 저장 등 시간이 걸릴 수 있는 작업에 대해 loading 상태를 표시해야 한다.

### 10.3 보안

사용자의 계정 정보와 개인 선호도 정보는 안전하게 저장되어야 한다. 비밀번호는 Supabase Auth가 관리하며, 프론트에는 access token과 refresh token만 저장한다.

Supabase service role key는 절대 프론트엔드에 노출하지 않는다. 백엔드 API는 access token을 검증한 뒤 사용자별 권한에 맞는 작업만 허용해야 한다.

### 10.4 개인정보 보호

음식 선호도, 알러지 정보, 식사 기록은 개인 정보에 해당하므로 사용자의 동의 없이 외부에 노출되지 않아야 한다. 모임 추천에서는 참여자의 선호도를 종합 결과로 사용하되, 개인의 민감한 제한 조건이 불필요하게 노출되지 않도록 한다.

게스트 사용자의 임시 계정은 메뉴 확정 이후 정리될 수 있으며, 모임 참여 기록에는 표시 이름만 남길 수 있다.

### 10.5 신뢰성

API 요청 실패, 세션 만료, 네트워크 오류, 백엔드 서버 오류가 발생해도 앱이 깨지지 않아야 한다. 사용자는 오류 원인과 다음 행동을 이해할 수 있어야 한다.

### 10.6 유지보수성

프론트엔드는 feature 단위로 화면과 상태를 분리한다. 백엔드는 API 라우터, 서비스, repository, validation 계층을 분리한다. DB 스키마와 API 문서는 실제 구현과 일치하도록 유지한다.

## 12. 설계 요약

먹픽은 개인과 모임의 메뉴 선택 과정을 단순화하기 위한 음식 추천 앱이다. 개인 추천에서는 카테고리 선호도, 태그 선호도, 메뉴 선호도, 알러지, 예산 조건, 최근 식사 기록을 반영한다. 모임 추천에서는 참여자별 선호도와 제한 조건을 종합하고, 모임 목적 적합도를 더해 모두가 수용하기 쉬운 메뉴 후보를 제안한다.

현재 구현 기준에서는 이메일 로그인, Google OAuth 로그인, 게스트 참여, 선호도 설정, 개인 추천, 모임 추천, 식사 기록, 메뉴 상호작용, 예산 조건, 최근 식사 중복 패널티가 MVP 핵심 기능이다. 식당 정보나 위치 기반 기능은 현재 MVP 범위에서 제외하고, 메뉴 자체를 추천하는 기능에 집중한다.
