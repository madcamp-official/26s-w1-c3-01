# 기능명세서 기준 구현 점검 및 보완 작업 목록

## 점검 기준

- 기준 문서: `docs/기능명세서.md`
- 점검 대상: 현재 구현된 프론트엔드 화면 흐름과 백엔드 API
- 목적: 기능명세서 대비 구현 완료 여부를 확인하고, 부족한 작업을 개발 체크리스트로 정리한다.

## 전체 결론

현재 구현은 MVP 핵심 흐름 대부분을 포함한다. 회원가입, 로그인, 게스트 시작, 선호도 입력, 개인 추천, 모임 생성, 모임 참여, 모임 추천, 모임 메뉴 확정, 식사 기록 생성과 조회는 동작 흐름이 마련되어 있다.

다만 기능명세서 전체 기준으로는 아직 부분 완료 상태이다. 특히 모임 정보 수정, 추천 결과 저장, 식사 기록 상세/수정/삭제, 메뉴 데이터 관리 CRUD, 모임 추천 공정성 로직은 추가 구현이 필요하다.

## 구현 완료 또는 대부분 완료

### U-01 회원가입

- 이메일, 비밀번호, 닉네임 기반 회원가입 API가 있다.
- 프론트 온보딩에서는 닉네임과 선호도 입력 후 계정을 생성하는 흐름으로 연결되어 있다.

### U-02 로그인 및 로그아웃

- 로그인 API와 로그아웃 API가 있다.
- 프론트에서 토큰 저장, 세션 복원, 로그아웃 시 로컬 세션 정리를 처리한다.

### U-04 게스트 시작

- 게스트 임시 계정 생성 API가 있다.
- 프론트에서 게스트 선호도 입력 후 모임 ID로 참여하는 흐름이 있다.
- 모임 메뉴 확정 시 게스트 계정을 정리하는 서버 로직이 있다.

### P-01 선호도 선택

- 카테고리, 태그, 메뉴 선호 점수를 저장할 수 있다.
- 프론트 온보딩과 선호도 관리 화면에서 선택 값을 API로 저장한다.

### P-02 제한 조건 선택

- 알러지 또는 제한 조건을 선택하고 저장할 수 있다.
- 추천 알고리즘에서 사용자 알러지와 충돌하는 메뉴를 제외한다.

### P-03 선호도 관리

- 저장된 선호도 조회와 전체 교체 저장 API가 있다.
- 프론트에서 기존 선호도를 불러와 수정할 수 있다.

### R-01 개인 맞춤 메뉴 추천

- 개인 선호도, 카테고리 점수, 태그 점수, 메뉴 점수, 식사 기록을 기반으로 추천 점수를 계산한다.
- 알러지 조건과 충돌하는 메뉴는 후보에서 제외한다.

### R-02 새로운 메뉴 제안

- 기존 식사 기록이 없는 메뉴를 새 메뉴 후보로 표시한다.
- 새 메뉴 포함 여부를 프론트에서 조정할 수 있다.

### R-03 추천 결과 제공

- 추천 랭킹, 점수, 카테고리, 추천 이유를 제공한다.
- 프론트에서 추천 결과를 카드 형태로 보여준다.

### G-01 모임 생성

- 모임 이름, 시간, 목적, 장소를 입력해 모임을 생성한다.
- 생성자는 자동으로 참여자로 추가된다.

### G-02 모임 ID 공유

- 모임 상세 및 목록에서 모임 ID를 표시한다.
- 프론트에서 모임 ID 복사 버튼을 제공한다.

### G-03 모임 참여

- 사용자가 모임 ID와 표시 이름을 입력해 모임에 참여할 수 있다.
- 게스트도 동일하게 표시 이름으로 참여한다.

### G-04 모임 정보 조회

- 모임 목록, 모임 상세, 참여자 목록, 추천 상태를 조회할 수 있다.
- 참여 전 모임 ID로 기본 정보와 구성원을 미리 확인하는 preview API가 있다.

### GM-01 모임 조건 반영

- 모임 참여자의 카테고리, 태그, 메뉴 선호도와 알러지를 추천 계산에 반영한다.

### GM-02 모임 선호도 분석

- 참여자별 선호 점수를 평균 방식으로 종합한다.
- 모임 목적 적합도도 점수에 반영한다.

### GM-03 모임 메뉴 후보 추천

- 알러지 충돌 메뉴를 제외하고, 모임 목적과 참여자 선호도를 합산해 메뉴 후보를 추천한다.

### GM-04 추천 결과 제공

- 추천 랭킹, 점수, 카테고리, 추천 이유를 모임 화면에 표시한다.

### GM-05 계산 참여자 선택

- 모임 상세에서 구성원 칩을 켜고 끄며 추천 계산에 포함할 참여자를 조정할 수 있다.

### GM-06 모임 메뉴 확정

- 모임 생성자가 추천 메뉴 중 하나를 확정할 수 있다.
- 게스트는 확정할 수 없도록 서버와 프론트에서 제한한다.

### H-01 선택 메뉴 저장

- 개인 추천 확정 시 식사 기록을 생성한다.
- 모임 메뉴 확정 시에도 식사 기록을 생성한다.

### H-03 식사 기록 조회

- 내 식사 기록 목록을 조회하는 API가 있다.
- 프론트에서 식사 기록 화면을 제공한다.

### H-04 만족도 기록

- 식사 기록 생성 시 rating을 저장할 수 있다.
- 추천 알고리즘에서 기존 식사 기록의 rating을 점수 보너스로 반영한다.

### M-02 메뉴 카테고리 관리

- 메뉴 카테고리 조회 API가 있다.
- 메뉴 조회와 추천에서 카테고리 정보를 활용한다.

### M-03 메뉴 속성 관리

- 메뉴 상세에서 매운 정도, 가격대, 칼로리, 태그, 알러지, 목적 적합도 정보를 조회한다.

### M-04 추천 제외 조건 관리

- 메뉴별 알러지 매핑과 사용자 알러지를 기준으로 부적절한 추천 후보를 제외한다.

## 부족하거나 부분 구현인 항목

### U-03 사용자 정보 관리

현재 상태:

- 내 정보 조회와 닉네임 수정 API는 있다.
- 프론트 프로필 화면에서는 닉네임 표시, 선호도 관리 이동, 로그아웃만 제공한다.

부족한 점:

- 프로필 화면에서 닉네임을 직접 수정하는 UI가 부족하다.
- 이메일 등 계정 정보 확인/수정 범위가 제한적이다.

필요 작업:

- 프로필 화면에 닉네임 수정 폼 추가
- 닉네임 중복/유효성 오류 메시지 정리
- 수정 완료 후 세션 프로필명 즉시 반영

### G-05 모임 정보 수정

현재 상태:

- `PATCH /meetings/:meetingId` 라우트는 있으나 실제 DB update 없이 요청 값을 그대로 응답한다.

부족한 점:

- 생성자 권한 확인이 없다.
- 모임 이름, 날짜, 시간, 장소, 참여자 수정이 DB에 저장되지 않는다.
- 프론트에서 모임 수정 UI가 없다.

필요 작업:

- `meetingRepository.update` 구현
- 생성자 권한 검증 추가
- 수정 가능한 필드 validation 추가
- 프론트 모임 상세에 수정 UI 추가
- 참여자 추가/삭제 또는 상태 변경 정책 확정

### H-02 추천 결과 저장

현재 상태:

- 모임 추천은 recommendation run 저장 흐름이 있다.
- 개인 추천 확정은 식사 기록 memo로만 남긴다.

부족한 점:

- 개인 추천 결과 후보 목록과 최종 선택 결과를 별도 추천 기록으로 저장하지 않는다.
- 추천 결과와 식사 기록이 명확히 연결되지 않는다.

필요 작업:

- 개인 추천 실행 결과 저장 테이블 또는 API 구현
- 추천 후보 목록, 점수, 이유, 선택 메뉴 ID 저장
- 개인 추천 확정 시 추천 기록과 식사 기록 연결
- 모임 추천 run과 확정 메뉴 저장 구조 점검

### H-03 식사 기록 상세/수정/삭제

현재 상태:

- 식사 기록 생성과 내 기록 목록 조회는 구현되어 있다.
- 상세 조회, 수정, 삭제 컨트롤러는 실제 DB 작업 없이 응답만 반환한다.

부족한 점:

- 상세 조회 권한 검증이 없다.
- 수정/삭제가 DB에 반영되지 않는다.
- 프론트에서 식사 기록 수정/삭제 UI가 없다.

필요 작업:

- `mealHistoryRepository.findByIdForUser` 구현
- `mealHistoryRepository.updateByUser` 구현
- `mealHistoryRepository.deleteByUser` 구현
- controller에서 실제 service 호출
- 프론트 식사 기록 수정/삭제 액션 추가

### M-01 메뉴 데이터 관리

현재 상태:

- 메뉴 목록/상세 조회는 구현되어 있다.
- 카테고리, 태그, 알러지, 목적 적합도 조회도 구현되어 있다.

부족한 점:

- 메뉴 등록, 수정, 삭제 API가 없다.
- 카테고리/태그/알러지/목적 적합도 관리 API가 없다.
- 관리자 권한 모델이 없다.

필요 작업:

- 1차 범위에서 "관리"를 조회로 한정할지, CRUD까지 포함할지 결정
- CRUD까지 포함한다면 관리자 권한 정책 추가
- 메뉴, 카테고리, 태그, 알러지, 목적 적합도 생성/수정/삭제 API 구현
- 관리자용 UI 또는 seed 기반 관리 방식 선택

### 모임 추천 공정성 로직

현재 상태:

- 참여자 평균 선호도와 목적 적합도를 기반으로 점수를 계산한다.
- 알러지는 후보에서 제외한다.

부족한 점:

- 특정 참여자에게 지나치게 불리한 메뉴를 막는 별도 로직은 약하다.
- 강한 기피 조건을 가진 참여자에 대한 최소 만족도 보장 로직이 없다.
- 선호도 충돌이 큰 경우 중립 메뉴를 선택하는 정책이 명확하지 않다.

필요 작업:

- 참여자별 메뉴 점수 계산 후 최저 점수 패널티 추가
- 강한 기피 점수 기준 이하인 메뉴 제외 또는 큰 감점
- 평균 점수와 최저 만족도를 함께 고려하는 정렬 기준 도입
- 추천 이유에 "공통 선호", "충돌 완화", "알러지 제외" 등 설명 추가

### 종교적 제한 및 기타 제한 조건

현재 상태:

- 알러지/제한 조건 테이블을 통해 일부 제한 조건은 표현 가능하다.

부족한 점:

- 종교적 제한, 기타 제한 재료가 별도 개념으로 분리되어 있지 않다.
- UI에서도 알러지 중심으로 표현된다.

필요 작업:

- 제한 조건을 allergy, religion, ingredient, custom 등으로 분류할지 결정
- DB seed와 UI 라벨 정리
- 추천 제외 로직에서 제한 조건 유형별 메시지 제공

## 우선순위 제안

### 1순위: 기능명세서와 실제 동작 차이가 큰 스텁 제거

- 모임 정보 수정 API 실제 구현
- 식사 기록 상세/수정/삭제 API 실제 구현

### 2순위: 추천 기록 저장 구조 보완

- 개인 추천 결과 저장
- 개인/모임 추천과 최종 선택 결과 연결

### 3순위: 프론트 기능 보강

- 프로필 닉네임 수정 UI
- 모임 정보 수정 UI
- 식사 기록 수정/삭제 UI

### 4순위: 추천 품질 개선

- 모임 추천 공정성 로직 강화
- 강한 기피 조건 처리
- 제한 조건 유형 분리

### 5순위: 메뉴 데이터 관리 범위 확정

- 메뉴 데이터 관리를 조회/seed 기반으로 볼지, 관리자 CRUD까지 할지 결정
- 결정된 범위에 맞춰 API와 UI 추가

## 검증 필요 사항

- Supabase 실제 환경변수 설정 후 전체 API 수동 테스트
- 회원가입 후 선호도 저장, 개인 추천, 식사 기록 저장 흐름 테스트
- 모임 생성, 참여, 추천 계산, 메뉴 확정, 게스트 삭제 흐름 테스트
- 알러지 조건이 추천 결과에서 실제로 제외되는지 테스트
- 생성자 외 사용자가 모임 메뉴 확정 또는 모임 수정을 시도할 때 차단되는지 테스트
